Transaction Lifecycle

From wallet to DAG

Every SIKKA transaction follows a deterministic 5-step lifecycle before joining the DAG.

Step 01
Build TX
Wallet selects inputs, outputs, and two parent tips from the current DAG frontier.
Step 02
Mine PoW
SHA3-256 nonce found to satisfy current difficulty bits. Prevents spam.
Step 03
Sign (ML-DSA-87)
All signers apply ML-DSA-87 signatures. Multisig: up to 16 parties sign collaboratively.
Step 04
Broadcast
Submitted to any node via HTTP API over Tor. Node validates and gossips to peers.
Step 05
DAG Entry
Valid TX appended to DAG. Referenced by future transactions — confirmation probability grows.
Finality confidence — references by later transactions
Consensus

Probabilistic Finality

SIKKA uses DAG-based probabilistic finality. As a transaction accumulates references from later transactions, the probability that it belongs to a valid ledger state approaches 1.

In a conflict (double-spend attempt), honest nodes apply a deterministic conflict resolution rule: the conflicting transaction with greater cumulative DAG weight wins. Attackers must outpace the entire honest network's transaction graph.

Conflict Resolution

When two transactions attempt to spend the same output, the one that accumulates more DAG weight — determined by the total PoW embedded in all descendent transactions — is accepted. The other is orphaned. All honest nodes converge on the same outcome deterministically.

Deterministic Convergence

Every honest node independently applies the same deterministic rules to the same DAG. There is no leader, no voting round, no BFT committee. The state emerges from the graph structure alone.
