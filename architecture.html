How the DAG works

Transaction Lifecycle

How a transaction enters the DAG

1

Select two DAG tips

The sender queries the network for current DAG tips — the most recent unconfirmed transactions. Two are selected as parents.

2

Build the transaction

Inputs, outputs, and the two parent references are assembled. Inputs must exactly equal outputs — no change, no fee.

3

Solve the PoW puzzle

A SHA3-256 nonce is found such that the pow_hash has the required leading zero bits. Difficulty is adaptive to network load.

4

Sign with ML-DSA-87

The transaction is signed with post-quantum ML-DSA-87. For multisig wallets, multiple signers collaborate before broadcasting.

5

Broadcast & propagate

The transaction is gossiped to peers. Each node validates and extends the DAG. The sender's transaction is now a tip.

Conflict Resolution

Probabilistic finality

Double-spends are not instantly rejected. Competing transactions race until accumulated work picks a winner — per UTXO, without blocks or miners.

1

Both accepted

If two transactions spend the same UTXO, both are gossiped and stored. No mempool gatekeeper.

2

Work accumulates

Each descendant adds weight: 2pow_bits per transaction. The branch with more cumulative work wins.

3

One survives

Once weight crosses the confirmation threshold, the canonical spend is settled. The loser is pruned.

Cumulative weight accumulation toward confirmation threshold (200).

Winner Selection
canonical_spend = argmax cumulative_weight(tx)
cumulative_weight(tx) = Σ 2pow_bits over all descendants
Tie-break: lower tx_id (lexicographic) wins
Confirmed when cumulative_weight ≥ 200

No global lock

Concurrent double-spends resolve by accumulated work at per-UTXO granularity.

Fast in practice

At base difficulty, finality settles in seconds under normal load.
