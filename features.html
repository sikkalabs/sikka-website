Core Protocol

Elegantly simple invariants.

SIKKA is built on a small set of elegant, composable rules. No complicated smart contracts, no global coordinator — just math.

Why SIKKA?

A scalable smart ledger. Not a blockchain.

Zero Fees

SIKKA has strictly 0 fees. When you send SIKKA, the exact amount arrives. No gas, no priority fees, no hidden costs.

Blockless Architecture

SIKKA is technically not a blockchain. It uses no Proof of Stake and no global Proof of Work race. Every transaction is its own accepted unit.

Parallel Scalable

Unlike blockchains that congest, SIKKA is parallel scalable. Because transactions process asynchronously, the more users interact, the faster it gets.

Core Protocol

Three core invariants.

Exactly Two Parents

Every non-genesis transaction references precisely two earlier transactions. A true directed acyclic graph, not a blockchain.

Exact Balance Only

Inputs must exactly equal outputs. There is no fee field. The sender transmits the full amount with cryptographic certainty.

Per-Transaction PoW

Each transaction mines its own SHA3-256 nonce, providing built-in spam resistance. Difficulty scales with network congestion.

Proof of Work

Spam resistance without fees.

Sikka has no gas and no miner fees. Instead, every transaction solves a SHA3-256 puzzle whose difficulty rises automatically when the network is busy.

Difficulty Formula
required_bits = BaseBits + floor(recent_count / 60) × 2
BaseBits = 2  •  BucketSize = 60 txs  •  BucketBits = 2
recent_count = transactions timestamped in the last 60 seconds

Normal use

At baseline load, ~4 hashes per payment. Negligible on any device.

Under attack

Each extra 60 txs/min adds 2 bits — 4× more work per bucket.

Not a fee — a throttle

There is no recipient and no money collected. Flooders pay exponentially more CPU as they raise network load.

Anti-selfish-mining

Tip commitment

PoW is bound to the live state of both parent tips at mining time. You cannot pre-mine a stockpile — if the DAG moves while you hash, your work is instantly stale.

PoW Input
pow_input = txID
|| parent[0].pow_hash
|| parent[1].pow_hash
|| nonce
pow_hash = SHA3-256(pow_input)
Cryptography

Post-quantum ready.

Sikka relies entirely on the ML-DSA-87 (Dilithium) signature scheme. We do not use any classical elliptic curves, ensuring maximum security against future quantum threats.

  • No classical fallbacks Pure post-quantum cryptography throughout.
  • Deterministic sync State verification is fully deterministic.
Transaction Spec
txid = SHA3-256(version || parents || inputs || outputs || ts)
pow_hash = SHA3-256(txid || pow_nonce)
witness = ML-DSA-87 signature(s)

Smart Wallets

Native n-of-n multisig.

In Sikka, standard wallets are inherently n-of-n, and multisig is elegantly abstracted as an m-of-n wallet. No complicated smart contracts required.

Any single transaction can be collaboratively built and signed by up to 16 different wallets together.

Networking

Privacy and interoperability.

Tor-Native

Privacy is built into networking from the start. Nodes seamlessly advertise onion addresses, communicate privately, and support managed Tor integration.

HTTP Federation

Sikka nodes expose a clean HTTP API with CORS support. Makes it easy to federate networks, build lightweight web clients, and integrate with web applications.
